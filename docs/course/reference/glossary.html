The five players who is in the room

Softphone both
The dialer running as JS (SIP.js + WebRTC) inside the Odoo browser tab. Speaks SIP over WSS for signaling and WebRTC/SRTP for audio. Your handset.
enterprise voip module config
The customer's Odoo DB: numbers, extensions, groups, queues, voicemail, call log, the buy-number wizard. Holds zero Telnyx or Wazo credentials — it asks the shop for outcomes (/api/phone_service/1/pbx/<op>). Your office.
IAP phone_service config
Odoo-hosted broker ("the shop"). The only holder of Telnyx + Wazo master keys. Buys DIDs, bills in IAP credits, provisions tenants.
Wazo PBX both
Self-hosted IPBX = Asterisk + Kamailio + rtpengine + Python microservices. One server, one isolated tenant per customer. The switchboard.
Telnyx both
The carrier (CPaaS). Owns the real phone numbers and the wire to the worldwide phone network (PSTN). The phone company.
Golden rules: the customer DB never talks to Telnyx (and never shows the word "Telnyx" in its UI) — and since the DB→IAP→WAZO refactor it never talks to Wazo either; the only direct customer↔PBX link is the browser's SIP/WSS media channel. In customer-facing code the PBX is called only "Odoo PBX". (Onboarding §2)

The two planes the spine of everything

Control plane (signaling) signaling
The negotiation: who is calling whom, ring it, what codec, hang up. Carried by SIP. Small text messages. Routes through proxies. (RFC 3261)
Media plane media
The actual voice: a stream of audio packets (RTP, encrypted = SRTP). Takes its own path, usually more direct than the signaling. (RFC 3261 separates the two.)
Config / management plane config
Not part of a live call: REST + webhooks that set up users, numbers, dial rules, and report what happened (CDR). Odoo↔IAP↔Wazo live here.

SIP signaling the language of "ring!"

SIP signaling
Session Initiation Protocol — the application-layer protocol that creates, modifies, and ends calls. Signaling only; it never carries audio. (RFC 3261)
REGISTER signaling
"I'm online — route my calls to me here." How the softphone tells the PBX where it is.
INVITE signaling
"Start a call to X." Carries an SDP body proposing codecs/IPs/ports.
180 Ringing / 183 Session Progress signaling
Provisional replies. 183 = early media (you hear ringback/announcements before answer).
200 OK / ACK signaling
Answered, and the answer confirmed. The call is now up; media flows.
BYE / CANCEL signaling
Hang up — after answer (BYE) or before (CANCEL).
SDP signaling
Session Description Protocol — the body inside SIP messages listing codecs, IP addresses and ports. It's how the two sides agree on the media details.
SIP proxy signaling
Forwards SIP messages without terminating the session; media flows past it. Routing + security. Kamailio's role.
B2BUA both
Back-to-Back User Agent — terminates one call leg and creates another, sitting in the middle of both signaling and media. Runs call logic (IVR, transfer, bridge). Asterisk's role.
Registrar signaling
The component that accepts REGISTERs and remembers where each user is. At scale this is Kamailio's job, not Asterisk's.

Media & WebRTC the audio itself

RTP / SRTP media
Real-time Transport Protocol — the audio packets over UDP. SRTP = the encrypted variant.
WebRTC media
The browser's umbrella media stack. Inside it: signaling rides SIP-over-WSS, keys are negotiated by DTLS, audio rides SRTP/UDP. "WebRTC audio" and "SRTP over UDP" are the same thing at different precision.
DTLS-SRTP media
The handshake (over UDP) that negotiates the SRTP encryption keys.
ICE / STUN / TURN media
NAT traversal: the algorithm (ICE) to find a working path, a public-IP lookup (STUN), and a media relay of last resort (TURN, needed for symmetric NAT).
Opus / G.711 media
Codecs. Opus = modern, browsers. G.711 (µ-law/a-law) = PSTN standard. Different codecs on the two legs ⇒ transcoding.
rtpengine media
Wazo's media relay: shuttles and (when needed) transcodes the RTP/SRTP audio. The component that actually touches your voice.
Early media media
Audio before the call is answered (ringback, "number disconnected"). Signaled by 183.
The "silent call" failure (connects, no audio) is almost always a media-plane problem: NAT, firewall, the wrong public IP in SDP, or a codec mismatch. Signaling works over TCP/443 like normal HTTPS; media needs the wide UDP RTP range (10000–20000) open. (Wazo session §7, §06-ports)

Kamailio · Asterisk · Wazo internals inside the switchboard

Kamailio signaling
High-performance SIP proxy / registrar / router. No media. Terminates browser WSS, handles registrations at huge scale, and is the hardened public-facing SIP front door. The edge.
Asterisk both
The telephony engine (B2BUA). Runs the dialplan, IVR, voicemail, queues, transfers — the call logic. Orchestrates media (delegated to rtpengine in Wazo). The brain.
PJSIP signaling
The modern SIP stack inside Asterisk (replaces the old chan_sip).
wazo-confd config
Configuration DB + REST API: source of truth for users, lines, extensions, dial rules, queues. Odoo configures Wazo through this.
wazo-calld config
Runtime call control (originate, transfer, hangup) via ARI.
wazo-auth config
Token issuance + multi-tenancy. The shop authenticates here with its master credential; every call it makes must explicitly declare its tenant scope (Wazo-Tenant header). Customers no longer receive Wazo tokens at all.
wazo-agentd config
The queue-agent runtime API: login/logoff/pause an agent, read logged/paused status. Distinct from confd's agent configuration.
wazo-webhookd config
Pushes event webhooks (call created/answered/ended, voicemail message created) — to the shop, authenticated by a per-tenant secret; the shop filters, renames and re-signs them for the customer DB. Wazo never calls the customer DB directly.
ARI / AMI / AGI config
Three ways to control Asterisk from outside. Stasis = the handoff from the dialplan to an external app (e.g. Odoo) over ARI — our v2 "live CRM data at call time" escape hatch.
Tenant config
One isolated customer space inside the one shared Wazo (own contexts, trunk, users). Isolation is logical, by tenant_uuid — not separate servers.
Dialplan signaling
The rules for what happens when a call arrives (time conditions, routing, queues, voicemail). In Wazo it lives in wazo-confd's DB. Decision: Odoo 20 Callflow compiles to confd config (not live interpretation) for v1.
CDR / CEL config
Call Detail Record / Channel Event Log — the "what happened" records that drive billing.

Telnyx the carrier's vocabulary

DID number config
Direct Inward Dialing — a public E.164 number (+3225551234) bought from Telnyx, stored in voip.did.number.
SCC signaling
SIP Connection / Credential Connection — Telnyx's trunking config. In the old Telnyx-direct world one SCC was shared by everyone (the spoofing risk that helped motivate Parrot); today the trunk runs Wazo↔Telnyx and customers are isolated per tenant.
CCA signaling
Call Control Application — inbound routing config (so a call to your DID comes back to us).
OVP config
Outbound Voice Profile — which caller-IDs and rate plans are allowed. A DID must be assigned here to be usable as caller-ID (fixes D51).
PAID / PPID / RPID / From signaling
Caller-ID SIP headers, in Telnyx's priority order PPID > PAID > RPID > From. In the old Telnyx-direct softphone, browser JS stamped P-Asserted-Identity onto the INVITE (the D35 fix — and a spoofing hole). In Parrot, caller identity is set server-side on the Wazo user's outgoing_caller_id; the headers are now the switchboard's business.
D-codes signaling
Telnyx hangup causes. D35 = caller-ID invalid; D51 = caller-ID unverified (an OVP config step, not a code bug).

Odoo · IAP our application layer

Extension config
Your internal desk number (3–5 digits, from 1000). voip.extension; routes to a user, a call group or a queue via destination_ref. The DID's only writable link — did.user_id is computed from it.
Call group config
voip.call.group — a set of extensions that ring together (Wazo group, ring strategy "all").
Queue config
voip.queue — a waiting room: callers hold, the PBX rings agents by strategy (ringall/linear/rrmemory/…) with priorities. Two layers: voip.queue.agent.source = declared intent (ordered users/groups); voip.queue.agent = materialized roster with live status. ⚠️ Unrelated to voip.queue.mixin, the softphone Next-Activities list.
Agent config
A Wazo agentd object — the "worker badge" queues ring, distinct from the Wazo user (the phone identity). One per Odoo user (res.users.voip_pbx_agent_id), agent number = extension number, shared across queues.
Voicemail box / message config
voip.voicemail — a numbered answering machine on the PBX (default number "90"+id; dial *98 for your own, *97<n> for box n). voip.voicemail.message — one received recording; metadata is pushed via webhook, the audio is pulled back through the shop.
Audio message / Voice config
voip.audio.message — a reusable audio asset (upload or on-prem Piper TTS) used as a greeting; transcoded in pure Python to telephony format (8 kHz mono 16-bit WAV). voip.voice — a Piper voice model, downloaded on demand from an allowlisted repo.
error_key config
The stable machine-readable key on every shop error (PhoneServiceError.error_key, e.g. not_found, insufficient_credits). Code branches on the key and renders translated text — never show or parse a raw server message.
IAP credits config
Odoo In-App-Purchase balance. Numbers cost credits/month; calls cost credits/minute. The shop's currency.
Requirement / Requirement group config
KYC paperwork some countries demand: textual, document, address, or action (live identity verification). Grouped per (country, number type) in voip.requirement.group. action requirements belong to one specific DID (did_number_id); the others are shared by the group.
Incall / Outcall signaling
Wazo contexts: incall maps DID → destination — a user, a group or a queue (inbound); outcall maps _X. / _+X. dialing → trunk (outbound).

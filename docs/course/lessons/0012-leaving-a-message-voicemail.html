Leaving a message

Numbered answering machines on the switchboard, greetings cooked down to telephone-grade audio, and messages that find their way home.

⏱ ~11 minutes🎯 Goal: trace a voicemail from beep to play button, and know the four models by heart🧩 Builds on: Lessons 5, 9, 11

The answering-machine room

Picture a back room in the switchboard building (your Wazo tenant, Lesson 8) full of old-school answering machines. Each machine has a number taped to it — box 9012 — and each plays its own greeting tape when a caller lands on it. When a machine blinks with a new message, the concierge (the shop, the IAP service) slips a note under your door: "new message on machine 9012, from this caller, this long". The note is just paper — to actually hear the tape, you walk back down with the concierge and fetch it. Hold that picture: the note is pushed, the tape is fetched.

The four models

All four are VoIP-admin gated, and the box follows the two-brains pattern you know from users and queues (Lesson 11): Odoo holds the intent, syncs create-or-update keyed on the stored PBX id, and voip_skip_pbx_sync breaks write-recursion.(onboarding §11)

The listen flow — from beep to play button

Walk it once in words. Wazo stores the message and its webhookd subscription (created at tenant provisioning) POSTs to the shop, authenticated by a per-tenant secret. The shop renames the event to global_voicemail.message_created and forwards it to /voip/api/event, HMAC-signed. Odoo verifies the signature and creates the message row idempotently: search first, create inside a savepoint, and if a concurrent delivery already won the race, catch UniqueViolation and re-search. If the message is non-empty, Odoo immediately pulls the recording back through the shop — base64-over-JSON, because there is no multipart anywhere in this pipe — and stores it as an attachment. The Messages page then shows an inline ▶ that streams /voip/voicemail/message/<id>.(onboarding §11)

THE BIG IDEA — push the metadata, pull the audio. Webhooks must stay small and safely retryable: the retry mechanism is literally "return 503 and let Wazo redeliver" — the shop stores nothing, so the payload has to be cheap to resend. Audio is big, and fetch-on-demand is naturally idempotent: if the pull fails, Odoo logs a warning, the row survives without audio, and the next redelivery (or a manual replay) heals it. No retry queue, no quarantine table — the design makes failure boring.

Exactly-once without a queue: why TWO unique keys?

Look at the pair of indexes on voip.voicemail.message. (event_id) catches the same event delivered twice — the normal case, since 503-driven redelivery resends the identical event. (voicemail_id, pbx_message_id) catches the same PBX message arriving under a fresh event id — say, a re-subscription or a second event name for the same message. Either key alone leaves a duplicate-row hole; together with the savepoint + re-search they make a textbook idempotent consumer in about 40 lines.(voip_voicemail_message.py:40-45, 84-128)

Greetings: cooking a tape down to telephone grade

A phone line only carries frequencies up to roughly 4 kHz — that is why phones sound like phones — so telephony audio is 8 kHz, mono, 16-bit WAV. Whatever the admin uploads as a greeting gets transcoded to that format in pure stdlib Python (struct + wave): stereo is downmixed by averaging channels, and the sample rate is converted by linear interpolation. No ffmpeg — deliberately zero external dependencies.(voip_audio_message.py:217-292)

The cooked WAV is pushed to Wazo's calld greetings API with the slot fixed to "unavailable" (of Wazo's unavailable/busy/name trio); on the shop side the upload is a delete-then-POST replace. The box itself syncs with attach_audio=True, so Wazo also emails the recording to the box's notification address.

Sync strictness is graded. Box sync failures raise — the admin must know, because Odoo and Wazo would silently disagree about what exists. Sound uploads and recording fetches only warn — they are deferrable, and a replay fixes them. Fail loud where state diverges; log where retrying later is fine. That is the project philosophy in one line.

Getting callers in — and dialing in yourself

How does a caller ever reach box 9012? Through a user's fallbacks (Lesson 11): if your no-answer or busy destination is a string that matches a synced box number (or *97 + that number), Odoo turns it into a typed voicemail destination for Wazo — {type: voicemail, voicemail_id, greeting: unavailable, skip_instructions}. skip_instructions suppresses Wazo's canned "please leave a message" prompt when a custom greeting exists, so callers hear only your tape. Dial-in codes: *98 is your own box (that is what the softphone's voicemail button dials), *97<n> reaches box n.(res_users_settings.py:179-210)

TTS that never leaves the building

Instead of uploading a file, an audio message can be generated: Piper TTS runs on the Odoo server itself — no text or audio leaves the database for generation. Voices are ONNX models downloaded on demand, only from the allowlisted Hugging Face repo, md5-verified and size-capped; every failure mode (piper not installed, voice not downloaded, empty output) is a clean UserError, never a stack trace. One honest status note: every audio asset also mirrors best-effort to the Wazo confd sound category "odoo" — this is forward-compat only, nothing on the PBX consumes those sounds yet, and deleting the asset leaves the Wazo copy behind as an orphan.(onboarding §11)

Gotchas to keep in your pocket. (1) Box numbers are only checked unique against other voicemails — not against user extensions in the same context; a colliding admin edit is possible. (2) user_voicemail.* events are forwarded by the shop but deliberately unhandled in Odoo — Odoo boxes are standalone, not attached to Wazo users, so their messages arrive via the global event only. (3) The recording download is issued under the confd URL prefix where upstream Wazo documents calld — tests and demo mode mock it, so if listen-to-message misbehaves on a live box, this is your first suspect.

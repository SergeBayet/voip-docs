Calling the outside world

The full outbound journey — browser to a real phone — and the one thing that trips it up: caller ID.

⏱ ~10 minutes🎯 Trace an outbound call + explain the D35 fix↩ uses Lesson 1 (planes) + Lesson 2 (lifecycle)
Start simple 🧒

You want to call a normal phone out in the world — say a customer's mobile. Think of it like mailing a letter:

  • You write the letter and drop it at the post office (that's Telnyx, our carrier — it has the trucks that reach every house/phone in the world).
  • Every envelope needs a return address. On a phone call, the return address is the caller ID — the "who's calling" your customer sees on their screen.
  • The post office only mails your letter if the return address is really yours. Put a fake one, and they hand it back.

That "is this return address really yours?" check is the whole drama of this lesson.

The journey, as a picture

Notice the players are exactly Lesson 1's: Kamailio (signaling edge) forwards, Asterisk (engine) runs the call and sends it out the trunk — the permanent "thick wire" between our PBX and Telnyx — and Telnyx bridges onto the worldwide phone network (the PSTN).

The return-address drama: caller ID

Here's what actually broke for us, and why. By default our browser softphone wrote the wrong return address — it stuffed its internal login name into the caller-ID slot, because that's what the SIP library does out of the box. Telnyx looked at it and refused. That refusal is Telnyx error D35.

❌ Before (D35)
INVITE   From: u42-clienta
Telnyx: "that's not a phone number I gave you." → rejected (D35)
✅ After (the fix)
INVITE   From: u42-clienta
P-Asserted-Identity: +18076958558  ←  a DID we own
Telnyx: "yes, that's their number." → accepted, shows +1 807 695 8558

The trick: a SIP INVITE can carry several caller-ID headers, and Telnyx trusts some more than others. This is their priority order:

Caller-ID headerTelnyx trustWho set it
PPID — P-Preferred-Identityhighest—
PAID — P-Asserted-Identityhigh ← we use thisour fix adds it
RPID — Remote-Party-Idmedium—
Fromlowest (the default)SIP.js, automatically

So adding P-Asserted-Identity with a real DID overrides the junk in From. Telnyx sees a number it owns, accepts the call, and that DID is what your customer sees.RFC 3325

🔌 Where this lives in our code. In user_agent.js, when the user is on the Odoo provider and has a DID, we attach a P-Asserted-Identity header to the outbound INVITE before sending it: 
const inviterOptions = { earlyMedia: true };
if (config.isOdooProvider && config.didNumber) {
  inviterOptions.extraHeaders = [
    `P-Asserted-Identity: <sip:${config.didNumber}@${config.pbxAddress}>`,
  ];
}
new SIP.Inviter(userAgent, makeUri(phoneNumber), inviterOptions);
Read it back to yourself: "set the caller ID" = "add one header line inside the INVITE." That's the whole fix — pure signaling plane. (Wazo Architecture Session §1)
⚠️ The catch (remember this for later). That header is written in browser JavaScript — which a determined user can edit. So today a user could assert someone else's DID and spoof their caller ID, made worse by everyone sharing one Telnyx connection. The real cure is to stop trusting the browser and stamp caller ID server-side in Wazo. That's one of the headline reasons for the whole migration — we'll trace it in a later lesson.

Bonus you'll hear about: there's a sibling error D51 ("caller-ID unverified"). That's not a code bug — it means Telnyx now reads your number fine, but that DID hasn't been authorised in the carrier portal (assign it to the Outbound Voice Profile). Same envelope, a paperwork problem instead of a code one.

Desks, teams & waiting rooms

A number is useless until the switchboard knows whose desk it rings — meet extensions, groups, queues and agents.

⏱ ~12 minutes🎯 Goal: know which record to touch (and which NOT to) for any routing change🧩 Builds on: Lessons 5, 8

Picture the office floor behind every phone number you sold in Lesson 10. Each person has a desk with a short desk number — that's an extension. A team can share one bell that makes every desk ring at once — a call group. For support, you open a waiting room where callers hold while staff take them in turn — a queue. And to work a waiting-room shift, each worker wears a badge — an agent. Same person, two roles: the desk is where you get called; the badge is how the waiting room calls you.

One writable link. You author exactly one assignment: did.extension_id. The DID's user_id is a stored compute from the extension's destination§8.2; the queue roster is derived from the sources; the Wazo objects are synced from all of it. When routing looks wrong, find which derived layer is stale — never write to it directly. (One DID per user is a deliberate, temporary UNIQUE constraint.)

A user is a user

Early builds rang a person through a hidden one-member "personal queue" on the PBX — that's history. Today an Odoo user maps to a real Wazo user with a WebRTC line: an actual account on the switchboard, not a trick§8.1.

  • Identity lives on res.users, in system-only fields: voip_pbx_user_id / _user_uuid / _line_id / _agent_id / _auth_password. The extension record keeps only pbx_extension_id.
  • Naming is server-side: the shop builds odoo-<login>-<odoo_id> plus a tenant-uuid suffix, and hands the SIP username back to be stored in the user's settings. Enterprise never invents Wazo names — deterministic names are also how orphans get re-adopted (below).
  • Personal routing = Wazo user fallbacks. Busy, no-answer (+ ring timeout) and always-forward rules are pushed from res.users.settings whenever a field in PBX_SYNC_FIELDS changes. A destination may be an E.164 number, an internal extension, a feature code like *97…, or a voicemail box — Lesson 12's whole topic.

Extensions: the desk number

An extension number is 3–5 digits, must not start with 0, is auto-allocated from 1000 up, and is globally unique (EXTENSION_NUMBER_RE, _allocate_numbers). Admins can create, renumber or remove a user's extension straight from user settings: pbx_extension_number is a compute/inverse on res.users.settings, and the inverse raises AccessError for anyone outside the VoIP-admin group.

Credentials outlive numbers. Releasing or suspending a DID never touches SIP identity — the user keeps their creds and stays registered. Only deleting the extension clears them: _unlink_pbx_resources unroutes DIDs, deletes the PBX objects, revokes the wakeup token, and wipes voip_pbx_auth_password + the settings' SIP creds. The extension, not the DID, owns the identity lifecycle§8.2.
Gotcha that cost real debugging time: the FK bypass. The DID's extension_id has an ondelete="set null" in the database. If we let Postgres do the nulling when an extension is deleted, Odoo's Python write() never runs — so the Wazo incall (the "this number rings that desk" rule) never gets torn down and leaks. That's why _unroute_did_numbers explicitly does routed_numbers.write({"extension_id": False}) before the unlink (voip_extension.py:260-266). Rule of thumb: any PBX-side side-effect must ride an explicit ORM write, never a DB-level cascade.

When provisioning breaks: healing (Lesson 8 recap, enterprise side)

  • Level-triggered checklist: _sync_user_provisioning / _filter_under_provisioned compare each user against "what fully provisioned looks like" — settings creds, PBX ids, wakeup token, incall ids — and re-drive only what's missing. Events trigger checks, not blind re-runs.
  • Safety nets: a 30-minute heal cron, plus a 60-second retrigger scheduled after any failure.
  • Separate savepoints: user provisioning and incall routing each get their own — a routing failure must never roll back freshly persisted PBX ids, or the retry would duplicate Wazo objects.
  • Orphan adoption: if Odoo crashed after creating Wazo objects but before saving their ids, the retry hits a conflict — recovery finds the orphan via its deterministic SIP username and re-owns it with re-issued secrets.

Queues & agents: intent vs roster

The waiting room uses a two-layer model. You declare who should staff it; the system works out who that actually is right now:

◆ voip.queue.agent.source — the INTENT

  • You write this. Ordered lines pointing at users or whole call groups.
  • Groups expand recursively (descendants too; recursion is a constraint violation).
  • Order matters: it becomes Wazo member priority. Duplicates? First occurrence wins.

● voip.queue.agent — the ROSTER

  • System-managed. One row per resolved user, rebuilt on every sync.
  • Carries Wazo runtime status: logged_out / available / paused / in_call.
  • Status is pull-only — the form's Refresh button, no cron. status_last_sync says how stale.

One Wazo agent per Odoo user (res.users.voip_pbx_agent_id), shared across every queue; the agent's number is simply the user's extension number. Ring strategy per queue: ringall (all badges at once), linear (top of the list first), rrmemory (round-robin that remembers where it stopped), leastrecent, fewestcalls, random — plus agent/queue timeouts, retry delay and a max-waiting-calls cap. (wrandom is deliberately hidden until per-member weights exist.)

The sync, in five hops (voip_queue.py:233-268):

  1. Resolve: _get_resolved_agent_users() expands the sources in order into a deduped user list.
  2. Per user: find-or-create their extension, sync it, then sync_agent(...) — create-or-adopt on the shop — storing the returned agent id on the user.
  3. Materialize the roster rows (stale rows unlinked, current ones upserted with sequence).
  4. Queue sync: sync_queue(..., agent_ids=[ordered wazo ids]); the shop reconciles Wazo queue members so priority = list position, and strips legacy user-type members.
  5. Ripple: editing a call group's members or an extension's destination re-resolves every queue that transitively references it — affected sets computed both before and after the write.

Deleting a queue removes its extension and the Wazo queue — but never the Wazo agents; they just lose that membership.

Honest status note. The full agentd surface — login, logoff, pause, per-queue login — is exposed on voip.pbx.service but has zero business callers today. It is plumbing for a future agent-presence UI. Whether a never-logged-in agent still receives queue calls is Wazo-side behaviour to verify when that lands — don't assume either way.
⚠️ Naming trap. voip.queue.mixin has nothing to do with PBX queues. It backs the softphone's "Next Activities" to-call list (today's phonecall activities). Worst name collision in the module — the mixin even carries a comment saying so (voip_queue_mixin.py:12-13).

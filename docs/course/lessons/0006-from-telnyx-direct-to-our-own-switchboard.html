From Telnyx-direct to our own switchboard

The old softphone rented everything from the carrier. Parrot bought the building. This is the why.

⏱ ~10 minutes🎯 Goal: argue the migration's trade-offs in a design discussion🧩 Builds on: Lessons 1, 2, 4, 5

Imagine your lemonade stand was a stall rented inside someone else's shop. Handy: the shop already has doors, locks, and a doorbell. But their doors, their locks, their doorbell — and every other stall shares the same entrance. You can't add a back door, you can't change who a bell rings, and if a neighbouring stall shouts your name at the door, the shopkeeper can't always tell it wasn't you. Parrot is the day Odoo stopped renting the stall and bought its own building: now we choose the locks (credentials), the doors (the SIP edge), and the doorbells (dialplan, queues, voicemail).

Two worlds, one drawing

Same browser, same carrier, same phone numbers. What moved is who stands at the door. In the old world the browser registered its SIP identity straight against sip.telnyx.com — the admin pointed the whole database at one WebSocket server and each user typed a username and secret by hand. In Parrot, the browser registers against our Wazo — the Kamailio edge from Lesson 2 — and everything else about the phone system became configuration we own.

The three wins, each anchored to an old-world failure

WinThe old-world failureThe Parrot answer
security Caller-ID rode along in SIP headers the browser composed (the P-Asserted-Identity hack). Any JS in any tab could stamp another customer's number — a spoofing class, not a bug. Caller-ID is now config on the Wazo user, written server-side through the shop. The client never composes its own identity, so there is nothing to forge.onboarding §8
isolation Every customer lived in one shared carrier account. Walls between customers were policy and hope. One Wazo tenant per customer, derived as UUID(hex=client_id) from the authenticated caller. The customer never names its tenant; cross-tenant access is impossible by construction, not by ACL diligence.onboarding §2
ownership Features ended where the carrier's API ended; call history was whatever the carrier's webhooks chose to report. Dialplan, queues, voicemail, call groups are ours to build. Native call events flow from Wazo's webhookd → shop → your DB, signed, deduped, in near real time.onboarding §9

What it cost — and what's still pending

Owning a building means maintaining a building. We now operate a PBX fleet: each Wazo box is Asterisk + Kamailio + rtpengine plus microservices (Lesson 2's trio, now our pager duty). And one corner is explicitly unfinished: the push path that wakes a closed browser so it can ring is an interim mechanism — the commit that introduced it literally says "webhooks (debut: waiting for Kamailio)", i.e. it stands in until the Kamailio-side infrastructure lands.c0faf70bde1

And Telnyx did not disappear — it was demoted, not fired. It still owns the phone numbers (the DIDs from Lesson 5), still runs the SIP trunk that connects our building to the worldwide phone network, and still bills per minute. We bought the building; the street outside is still theirs.

The layering that made it safe: DB → IAP → WAZO

Owning a PBX raises a scary question: does every customer database now hold keys to our switchboard? The answer is a hard no, and it's the most important design move of the migration. Early Parrot let enterprise talk to Wazo's config API itself — a full Wazo client lived in the customer's codebase. Then one commit deleted all ~4,600 lines of it: the ConfdClient, every command class, the lot.9f090a1757c Now enterprise asks the shop for outcomes ("sync this user's extension", "route this number") and the shop — sole holder of the Wazo master token — runs the choreography, pinned to your tenant.

The migration wasn't just moving servers — it removed a whole credential class from every customer database. No Wazo URL, token, or tenant id exists in enterprise code anymore. A compromised customer DB can annoy the shop; it cannot touch the switchboard — or the neighbours.
⌨️ Where this lives in our code
  • voip/models/pbx_service.py — the entire enterprise↔PBX surface: every method is one line, self._api().pbx_call("<op>", **kwargs) → POST /api/phone_service/1/pbx/<op>.
  • voip/data/voip_provider_data.xml — the "Odoo Phone Service" provider: ws_server = wss://wazo-1.voip.test.odoo.com/ws. This is the one address the browser dials directly — compare with the old voip.wsServer pointing at the carrier.
  • iap_services/phone_service/models/phone_service_pbx_tenant_ops.py — the shop side: the _op_* methods holding the full Wazo choreography, tenant-scoped, lookup-before-create.

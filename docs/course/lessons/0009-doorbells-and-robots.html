Doorbells & robots

How news travels between four machines that all crash sometimes — and why the answer is never a new queue table.

⏱ ~12 minutes🎯 Goal: apply the reliability playbook (sign, dedupe, 503, poll, heal) to any new event🧩 Builds on: Lessons 5, 7, 8
Start with the letterbox. 🏠 A doorbell ring is gone the moment it stops — if you were in the shower, that news is lost forever. Registered mail solves this differently, and notice who does the work: your house does not keep a warehouse of half-delivered letters. The postman keeps the letter and comes back tomorrow, and the day after, until you sign the receipt. You do exactly three things: check the wax seal (is this really from who it claims?), check "did I already get this one?" (duplicates happen), and sign — or refuse to sign, which just means "ring me again". Parrot's entire event system is that postal contract in HTTP: signing = answering 200, refusing = answering 503, and the sender keeps the letter.

The event geography — every doorbell on the street

Read the picture left to right. The carrier and the PBX both push news to the shop (number bought, call started, call ended, cost known). The shop cleans it up — renames Wazo events to a stable vocabulary, drops call legs you don't care about, applies the billing markup — and forwards it to one single URL on your database: /voip/api/event. Behind that one door, a plain dispatch on event_type hands each letter to the model that owns it. One door means one place to check seals, one place to log, one place to debug.

Three doors, three locks — and why each

Each hop uses a different signature scheme. That's not inconsistency — each lock fits its door:§14

DoorLockWhy this lock, here
Telnyx → shopEd25519 signature over timestamp|body, ±300 s tolerancePublic internet. The carrier publishes one public key; anyone can verify, nobody can forge, and the timestamp window kills replay attacks.
Wazo → shopPer-tenant secret in the body, constant-time compareAn intra-infrastructure hop over TLS. The secret (minted at tenant provisioning) both authenticates the sender and identifies which tenant is talking.
Shop → your DBHMAC-SHA256, key derived from client_secretThe shop stores only a hash of your secret, so it can't use the secret itself as a key — it derives a purpose-bound key once, at registration, and stores that instead.
The bug that motivated "derived". The shop→DB signing key used to be built from client_id. But client_id travels in every request your DB makes — it is effectively public. Anyone who saw one request could forge "signed" events into your database. The fix (commit 39382d2f778): derive the key from the secret — HMAC(client_secret, "iap-phone-service-webhook-signing"). Rule of thumb: a signing key must come from something that never crosses the wire.

The fourth arrow — your DB calling the shop (sync_queue, buy_number…) — isn't a webhook at all: it's an ordinary API call authenticated with client_id + client_secret on every request. Keep the two directions distinct in your head; the sorter below will check.

"Did I already get this letter?" — dedup is a constraint, not an if

Refusing to sign means the sender rings again — so every event can arrive twice, and your handler must treat a replay as a no-op. Parrot's answer is mechanical and identical at every hop:

  • Deterministic ids. The shop names Wazo events event_id = call_id:event_name — a fact has one name, so its redelivery collides with its first delivery. A random UUID per delivery would make every replay look new.
  • A UNIQUE constraint at every table that stores events: the shop's webhook-event log, enterprise voip.pbx.event, voip.call.cost.line, and voip.voicemail.message (that one twice — by event_id and by PBX message id). Payments too: UNIQUE(reference_type, reference_id) means a replayed cost event can never charge twice.
  • Race-safe insert. A cheap search_count pre-check, then create inside a savepoint, catching UniqueViolation as "already seen" (voip_pbx_event._register_event). Two workers processing the same replay concurrently both survive; exactly one wins.
Every dedup is a constraint, not an if. An if seen: return check has a race window between the check and the write. A database UNIQUE index has none — the database itself is the referee. If you catch yourself writing a "have I processed this?" flag, reach for a constraint instead.§15

When you can't finish: refuse to sign

Say a call.cost event arrives at the shop before the lifecycle events that identify whose call it was — the handler literally cannot attribute it yet. What now? The playbook says: answer 503 and store nothing. Telnyx keeps the letter and rings again in a few minutes, by which time the correlation row exists and the same handler succeeds. The shop does the same to Wazo webhookd, and your DB's ack (or missing ack) does the same to the shop: not acknowledged → the shop answers 503 upstream → the whole letter is re-sent later. The retry queue always lives at the sender — that's the postal contract, and it's why no box in the diagram has a dead-letter table.

The silent-200 anti-pattern. A 200 tells the sender "delivered — never send this again". So a handler that hits a problem, logs a warning, and returns success anyway hasn't handled the error — it has destroyed the only retry the system had. For a billing event that is unrecoverable revenue loss, and the only trace is a log line. Smell test: any _logger.warning(...) immediately followed by a successful return on a path that was supposed to do real work.

And the client-side mirror: push what you can, poll what you can't. A 503 protects you when a webhook arrives broken — but some webhooks never arrive at all (your server was down for an hour; a network path ate the POST). So every push has a slow polling fallback: an hourly cron re-asks the shop for number statuses, and another hourly cron re-tells the shop where your doorbell URL is. The poll is deliberately dumb and slow — it's the residue, never the design. If you find yourself building a clever fast poll, first ask: is there a push I'm ignoring?

The robots — who heals what

Nothing here trusts a single HTTP call to succeed; every async path has a robot that re-drives it. The six most instructive (full table: onboarding §15):§15

SideRobotEveryWhat it heals
🏪submit ordering1 minauthorize credits + send number orders to Telnyx
🏪resume PBX provisioning5 minlay the next tenant brick; mark failed after 24 h stuck
🏪recover payments10 minretry charges on exponential backoff (30 min·2ⁿ, cap 1 day; ≥5 tries ⇒ delinquent)
🏪reconcile client state1 mindrive Telnyx routing to match the disabled intent
🏠heal provisioning30 minre-drive under-provisioned active DIDs against the checklist
🏠link orphan cost lines30 minre-match cost lines that arrived before their call record

Notice what makes robots safe to run forever: every step is idempotent (lookup-before-create, adopt orphans on conflict), every webhook they back up has a polling fallback, and every "only once" rule is a constraint. A robot that re-runs a finished job must change nothing — that's the test for adding one.

⚠️ The cautionary tale — the quarantine that got reverted. Once, an unattributable call.cost event looked like it deserved real machinery: the team built a quarantine model + retry cron + escalation email for events that couldn't find their client. It worked. It was also reverted as over-engineered: three new moving parts, each needing observability and maintenance, to handle a rare case for which the upstream already had a retry queue. The fix that survived is one line of behavior: answer 503 and let Telnyx redeliver. The team's philosophy, verbatim from its engineering notes: fix what is broken; do not harden what merely could break — and when a failure is silent, the cheap reusable fix is logging the reason well, not building infrastructure around it. Before you add a table, a cron, or an alert: who already holds this letter?

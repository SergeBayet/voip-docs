Odoo becomes a phone company

The four machines behind a ringing browser tab — and the golden rules that keep them honest.

⏱ ~10 minutes🎯 Goal: name every player and every rule from memory🧩 Builds on: nothing — start here

The lemonade stand gets a phone line

Imagine you run a lemonade stand and you want a real phone number — one your customers can dial from anywhere in the world. You know nothing about phone companies, and you'd like to keep it that way. So you walk into a shop and say: "I'd like a phone number in Belgium, please."

The shop buys the number from a phone company that owns millions of them. It also rents you a desk in a big switchboard building and tells the phone company: "when this number rings, send the call to that building." The switchboard knows your desk number, and when a call arrives it rings your handset.§1

That is Parrot, Odoo's new VoIP era, told with toys. Your Odoo database is the lemonade stand; the handset is a web browser tab. You bought a real phone line without ever meeting a phone company.

The one-breath version: Odoo is becoming a phone company that lives inside your Odoo database. You click a button, pick a number, pay with credits — and from that moment, when somebody dials that number, your browser rings.§1

The golden rules

The diagram above is really a list of arrows that are missing. Each missing arrow is a rule, and each rule has a reason. Learn the reasons — they're what you'll defend in code review.§2

Rule 1 — The customer database never talks to Telnyx

Not one HTTP call. It never even shows the word "Telnyx" — in customer-facing code the whole thing is just "Odoo PBX". Why: the carrier is an implementation detail. If it lives behind the shop, Odoo can renegotiate or swap carriers without touching a single customer database — and the carrier's API key never has to exist in thousands of customer databases.

Rule 2 — …and it never talks to Wazo either

The customer database holds zero Wazo URLs, tokens or tenant ids. Instead of giving the switchboard instructions, it asks the shop for outcomes: "make this user's desk exist" (sync_user_extension), "wire this number to that desk" (sync_incall) — each one a POST /api/phone_service/1/pbx/<op>. The shop runs the multi-step Wazo choreography server-side. Why: the choreography lives in exactly one place, upgraded centrally, and the master token stays home. If you ever find enterprise code building a Wazo URL, it's a regression.

Rule 3 — The one carve-out is media

Look back at the diagram: one arrow does cross from the customer's side straight to Wazo — the softphone. The browser registers SIP-over-WSS to Wazo with its own per-user credentials. Why: your voice is a live stream, dozens of packets per second in both directions. A request/response JSON API can carry "please create a desk"; it cannot carry a conversation. So paperwork goes through the shop, sound goes direct.

Rule 4 — Tenancy is structural, not polite

Each customer gets their own locked floor in the switchboard — a tenant. Here's the trick: the floor number is derived from who's asking (tenant = UUID(hex=client_id)), so the caller never gets to name a tenant at all. And inside the shop, every call to Wazo must explicitly declare tenant_uuid=… or master_scoped=True — declaring neither, or both, is a hard ValueError. Why: you can't sneak onto someone else's floor if there is no field for "which floor" in the first place.

Rule 5 — Everything async is a cron retrying idempotently

Buying numbers, billing, building your floor — anything that spans machines is driven by little robots (ir.cron jobs) that retry, and every step is written so that running it twice is safe. Why: networks fail mid-way, guaranteed. Retry-safe steps are the honest alternative to pretending "exactly once" exists.

⚠️ Watch out: demo mode proves almost nothing. One system parameter (voip.test_mode) short-circuits everything at the very first layer: get_phone_service_api() returns a demo class with canned answers, PBX ops echo fake ids, credits are a fake 500. A green demo run exercises none of the shop, Telnyx, Wazo, signatures or billing — it proves only the UI flows. Don't let it lull you.§16
🔩 Where this lives in our code
  • enterprise/voip/ — the whole customer side: models, wizards, and the softphone under static/src/.
  • enterprise/voip/models/phone_service_api.py — the HTTP client to the shop. pbx_call(name, …) posts to /api/phone_service/1/pbx/<name>; get_phone_service_api() at the bottom is the demo-mode switch.
  • enterprise/voip/models/pbx_service.py — the named-op facade: every method (sync_user_extension, sync_incall, …) is exactly one /pbx/<op> call. This is Rule 2, as a file.
  • iap-apps/iap_services/phone_service/ — the shop. models/wazo_api.py is the transport that enforces Rule 4's mandatory scoping; utils/telnyx_client.py is the only code anywhere that speaks to Telnyx.
